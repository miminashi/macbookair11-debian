# ハングアップ対策レポート群の方法論監査レポート作成プラン

## Context

MacBook Air 11" (Early 2015) / Debian 13 の suspend ハング問題について、2026-05-10 から 2026-07-01 まで約 2 ヶ月・**29 本** のレポートが蓄積されている (①初期 S3 ハング期 7 本 + ②S3 復活・soak 期 9 本 + ③s2idle BT-VPN 切り分けキャンペーン 13 本。mozc 等ハング無関係のレポートは対象外)。ユーザの依頼は「過去のハングアップ対策関連レポートを読み、これまでのアプローチに問題がないか・見落としているポイントがないかを調査してレポートにまとめる」こと。

調査 (Phase 1) は完了済み。3 つの Explore エージェントで全 29 本を精読し、時期別 (①初期 S3 ハング期 5/10-6/03、②S3 復活・soak 期 6/08-6/27、③s2idle BT-VPN 切り分けキャンペーン 6/28-7/01) に方法論監査サマリを取得した。本プランはその所見を **1 本の監査レポート** にまとめる作業計画。

**コード変更・実機操作は一切なし**。成果物はレポート 1 本 (+ 添付プランファイル) のみ。

## レポート構成 (章立て) と盛り込む主要所見

### 1. 概要 (通読用)
- **細かい事実・数値は省き、平易な日本語で全体像を通読できるセクション** (ユーザ指定)。
- 内容: この 2 ヶ月で何が起きたか (S3 ハング → パラメータ 3 連敗 → s2idle 移行 → hibernate 修理 → S3 復活挑戦と no-go → s2idle BT-VPN ハングの切り分けキャンペーン) の物語としての流れ、監査して分かったこと (どこまでが確かで、どこからが未確定か)、アプローチの主な問題点 3〜4 点、見落としの要点、次に何をすべきか — を専門用語を最小限にして数段落で記述。
- 詳細は以降のセクションに委ね、このセクションだけ読めば全体像がつかめる粒度にする。

### 2. 前提・目的・監査の方法
- 監査の目的 (アプローチの問題点・見落としの洗い出し)、対象 29 レポートの 3 期区分。
- **監査の方法 (再現方法相当、CLAUDE.md ルール)**: 対象レポート全文の読解ベース監査であり **新規実機実験は行っていない** こと、3 エージェントによる時期別精読 + 原本 spot-check という手順を明記。
- 対象環境の要約 (macbookair2015.lan / Debian 13 / kernel 6.12.x 系) を簡潔に記載。

### 3. 確立された事実 (bedrock) と未確立事項の峻別
- **堅い**: (i) s2idle + BT-PAN + VPN + 手動 lid close で dpm_suspend 段の無音永久ハングが 3 セッション独立再現 (063543 3/10, 043251 1/20, 102907 1/26)。(ii) hibernate 系の修理 (BTP 経路のソースレベル確定 + 実 3% 発火観測)。(iii) S3 deep の待機電力優位 (~0.06–0.10 W、ゲージ非依存論拠)。(iv) 真の s2idle 初実証 (021019、ロールバック不完全の発見込み)。
- **未確立**: 機序 (H1–H7 いずれも未決着)、必要条件の精密化 ((b'') は post-hoc、Fisher p≈0.11 で有意未達)、base rate (30% vs ~5% で 3 倍以上乖離)。

### 4. アプローチの系統的問題点 (監査の本体)
1. **陰性証拠 (N/N clean) 依存の構造**: 低頻度間欠故障に対し「hang 0 = 効いた」に頼らざるを得ず、base rate 不安定 (30%/5%) のもとで (1-p)^N が 21%〜2e-5 まで振れる。102907 は heterogeneity を自認しつつ pooled 値を Fisher に投入しており、自己批判と統計処理が両立していない。
2. **valid cycle 定義の 7 段変遷と旧結論の無効化連鎖**: 手動 IKE 端点確認 → per-cycle PRE → (実質検証なし=200520) → xfrm count → +source-IP → +ping_running → +wl 状態。特に 200520 の「32/32 clean = 0.90^32≈3.4% で確定ライン」は cycle 2–32 が VPN inactive で完全無根拠と判明 (030349)。S1/S3 系の機序ラダー結論群が数日間、無効な valid 定義の上に積まれていた。
3. **AC 偏重・実故障モード未検証の温存**: 歴史的故障は battery / lid-close / STH なのに、「健全」証拠 (S3 復活期の 21/21・7/7、キャンペーン全 13 本) はほぼ AC。021019・063543 で「最も切り分けたい実条件」と明記されながら **battery/STH × BT-PAN+VPN は全 13 本を通じて一度も検証されていない**。6/27 の battery ハング (#4) は事実上予見可能だった。
4. **多変数同時変更の反復**: 5/31 の s2idle 化 (3 変数同時)、111259 (peer+経路)、S1 (btusb 除去+BT-PAN teardown)、130206 (rmmod wl = WiFi 機能+PCI suspend chain 同時除去) と、同型の「同時に複数を除去して分離不能」問題が繰り返されている。
5. **単発ハングへの過剰帰属と post-hoc パターン適合**: 043251/102907 は各 1 hang。6 セッション横断の「wl-loaded-AND-radio-off」分離は peer/hook/traffic/ping/駆動方式/N が全共変する 6 点への後付け適合で、keystone (061553) は実質 N=1 セッション (0/30 = 偶然 21%)。6/27 の「真因=BT-PAN」heading 昇格 (約 1 時間後に自己反証) も同種の踏み外し。
6. **観測装置の非中立性という原理的トレードオフ**: 70-h4-probe / 58-snapshot-only hook が suspend entry で実仕事 (書込+sync) をするため race を変えうる (candidate (c))。「素の 063543 条件を再現するには hook を外す必要があるが、外すと valid gate も失う」構造から脱出できていない。
7. **機序特定の構造的限界と S4 未着手**: journald flush の性質上、入眠/復帰 (α/β) は現行手法で永久に判別不能。102907 で pre snapshot が hang 直前 3 cycle で identical = 観測可能 state から hang は予測不能と判明済み。機序決着の唯一の出口 **S4 (DPM_WATCHDOG 自前カーネル) は全期間を通じて未実施**。
8. **初期 (5 月期) の教訓**: 効果測定 N=0 での恒久設定積み増し (i915/applespi/pcie_aspm 3 連敗)、弁別力の低いシグネチャ (最終行の揺れ) を仮説根拠に使用、rtcwake 検証が本番故障モード (lid) を再現しない構造的ギャップ。ただし 6/01 以降は対照実験へ質的に改善しており、時系列で方法論は明確に向上している — この点は公平に記す。

### 5. 見落とし・未実施の手 (優先順位付き)
- **レポート内で言及済みだが未実施**: S4 (DPM_WATCHDOG)、S2 (xfrm flush)、S5 (URB drain timeout patch、設計欠陥指摘のみ)、wl-unload N=60+ 拡大、btusb-removed arm の WiFi-off 再試行、battery/STH × BT-PAN+VPN、suspend 突入時の xfrm 残留確認。
- **レポートに言及がない盲点** (今回の監査で新規指摘): (i) 無介入・素条件での base rate 確立用 long soak が構造的に欠落、(ii) pstore/ramoops 常時 dump 未設定 (S4 依存を減らせる)、(iii) `/sys/power/pm_test` による段階分離が未検討、(iv) VPN 実装依存性 (WireGuard/OpenVPN で xfrm 特異性を切り分ける手)、(v) 非 Apple peer (Android テザリング)、(vi) btusb autosuspend 無効化 (`usbcore.autosuspend=-1`、ゼロビルドで可能)、(vii) mainline xfrm fix の backport 試行、(viii) α/β 直接分離 (lid close + RTC 先行 wake) が計画のみで未実行。
- 各項目に「何を判別できるか」「コスト感」を付す。

### 6. 総合評価と推奨
- 総括: 現象の存在 (bedrock) は堅いが、機序と必要条件は統計的に未確立。自己訂正能力 (030349 の confound 発覚、6/27 の反証受容など) は高く、監査上の評価は「誠実だが、陰性証拠依存と AC 偏重という 2 つの系統的バイアスが残る」。
- 推奨の次の一手 (優先順): ①base rate 確立 (素条件 long soak または既存 hang セッション条件の N 拡大)、②battery/STH 実条件セル の充填、③S4 DPM_WATCHDOG (機序決着)、④ゼロコスト系 (autosuspend=-1、ramoops)。

## 執筆時の正確性担保

- Explore サマリの記述を鵜呑みにせず、レポートに引用する具体的数値・主張 (例: 200520 無効化、(b'') の 6 セッション分離、5/56 pooled、0.7–0.8 件/週 baseline) は **原本レポートの該当箇所を grep/Read で確認してから記載する** (130206 の結論部・次の手は確認済み、整合)。
- メモリの機序ラダーファイル (`s2idle-btvpn-hang-mechanism-ladder.md`) とレポート本文で記号 ((a)-(e), H5 の再利用) に不一致があるため、レポートでは **どのレポートの記法か明示して引用する**。
- 監査レポート自身が新実験をしたかのような誤解を避けるため、冒頭に「本レポートは読解ベースの監査であり、新規実機実験は行っていない」と明記。
