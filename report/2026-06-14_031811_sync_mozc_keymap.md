# mozc 日本語入力キーマップを開発機に同期

- **実施日時**: 2026年06月14日 03:18 (JST)
- **対象機**: macbookair2015.lan (MacBook Air 11" Early 2015, Debian 13 trixie)
- **同期元**: 開発機 (このリポジトリが置かれ Claude が動作しているマシン)

## 添付ファイル

- [実装プラン](attachment/2026-06-14_031811_sync_mozc_keymap/plan.md)

## 前提・目的

- 背景: 運用対象機 macbookair2015 の「日本語入力中のショートカットキー割当」を、
  開発機 (このコンピュータ) と一致させたいという要望。
- 目的: macbookair2015 の mozc キーマップを開発機と一致させる。
- 前提条件: 両機とも ibus-mozc を使用し、ssh で実機へ非対話アクセスできること。

| 項目 | 開発機 (同期元) | macbookair2015 (対象) |
|---|---|---|
| IME | ibus-mozc **2.29.5160.102+dfsg-1.4** | ibus-mozc **2.29.5160.102+dfsg-1.4** (同一) |

| 稼働中 IME | ibus-daemon + ibus-engine-mozc + mozc_server | 同左 |
| 設定ファイル | `~/.config/mozc/config1.db` | 同左 |

- ssh 接続: `ssh miminashi@macbookair2015.lan` (作業時サンドボックス無効状態で疎通)

## 調査で判明した事実

### mozc キーマップの格納構造

- 日本語入力中のショートカット割当は `~/.config/mozc/config1.db` (**plaintext protobuf**) に格納される。
- 関連フィールド:
  - **field 41 = `session_keymap`** (enum: `NONE=0` / `CUSTOM=1` / ATOK=2 / MSIME=3 / KOTOERI=4 …)
  - **field 42 = `custom_keymap_table`** (`status<TAB>key<TAB>command` 形式の TSV 文字列)
- `history.db` / `segment.db` / `.encrypt_key.db` 等は学習履歴・暗号化系で機種固有 → 同期対象外。

### 両 config1.db の差分

protobuf としてフィールド単位で厳密比較した結果、**メタデータ (field 1: タイムスタンプ) を除く
差分は field 41・42 の 2 つだけ**で、他 47 フィールド (記号入力 field 54、文字形フラグ
field 80–90 等) はバイト単位で一致していた。

| field | 意味 | 開発機 (同期前) | macbookair2015 (同期前) |
|---|---|---|---|
| 41 | `session_keymap` | `0` (NONE) | `1` (CUSTOM) |
| 42 | `custom_keymap_table` | 5095 bytes (ことえり/MS-IME 風: Option キー・`Ctrl ;`/`:`/`'` 多用) | 4875 bytes (独自配列: `Ctrl j/k/l` 等の vim 風) |

→ キーマップ以外の設定は既に同一のため、開発機の `config1.db` を丸ごとコピーすることは
「キーマップ (field 41/42) だけを移植」することと等価で、副作用がない。

### 【要注意】開発機の実 live キーマップは未確定

- 開発機は `session_keymap=NONE(0)` でありながら 5095 バイトの独自カスタムテーブルを保持している。
- mozc で `NONE` が「内部既定キーマップに解決される」のか「格納テーブルを使う」のかは
  **本環境からは断定できなかった**。すなわち、**開発機で実際に効いているキーマップが
  その独自テーブルなのか mozc 既定なのかは未検証**。
- 今回保証したのは「両機が config 上バイト同一になった = 挙動が一致する」ことのみ。
  field 41・42 の両方をコピーすれば `NONE` の内部解釈に関わらず両機は同一挙動になるため、
  「一致させる」という目的には NONE の意味解決は不要だった。
- ただし「具体的にどのキーマップが live か」は本作業の保証範囲外である点に注意。

## 実施内容 (再現方法)

順序が重要: バックアップ → コピー → 即時 `pkill` (稼働中サーバが旧設定をディスクへ
書き戻すレースを避けるため)。すべて開発機の shell から ssh 越しに実行。

1. **対象機の現行設定をバックアップ** (タイムスタンプ付き)
   ```bash
   ssh miminashi@macbookair2015.lan \
     'cp -a ~/.config/mozc/config1.db ~/.config/mozc/config1.db.bak.$(TZ=Asia/Tokyo date +%Y%m%d_%H%M%S)'
   ```
   → `~/.config/mozc/config1.db.bak.20260614_030654` を作成。

2. **開発機の config1.db を対象機へ転送**
   ```bash
   cat ~/.config/mozc/config1.db | \
     ssh miminashi@macbookair2015.lan 'cat > ~/.config/mozc/config1.db'
   ```
   → 5406 バイト (開発機と同サイズ) で書き込み完了。

3. **mozc サーバ・エンジンを再起動して反映**
   ```bash
   ssh miminashi@macbookair2015.lan 'pkill -x mozc_server; pkill -f ibus-engine-mozc'
   ```
   - mozc_server は起動時のみ config1.db を読むため再起動が必要。
   - ibus-daemon は稼働継続。次回 IME 利用時に ibus-engine-mozc → mozc_server が
     遅延再起動し、新 config を読み込む (ログアウト不要)。
   - 補足: この ssh はエンジンプロセス kill の副作用で **exit code 255** を返したが、
     コピー・kill とも正常完了 (下記検証で確認済み)。

## 検証

適用後に対象機の config1.db を再取得し、protobuf をパースして照合:

- `session_keymap` (field 41) == `0` … 開発機と一致 ✓
- `custom_keymap_table` (field 42) … 開発機と **バイト完全一致** (5095 bytes) ✓
- **config1.db はファイル全体が開発機とバイト同一** (`cmp` で 5406 バイト完全一致、
  内部 last_modified タイムスタンプを持つ field 1 も含めて一致) ✓
  - コピー直後に mozc_server を kill したため、旧サーバによるタイムスタンプ書き戻し
    (前掲の「書き戻しレース」) が起きず、メタデータまで同一になった。
- 適用後の mozc_server / ibus-engine-mozc は終了済み、ibus-daemon は稼働継続を確認。

**実打鍵の最終確認はユーザに委ねる**: 対象機の GUI で実アプリに日本語入力し、変換中の
ショートカット (Space で変換、`Ctrl ;` 等の割当) が開発機と同じ挙動になるかを確認する。
反映されない場合はデスクトップセッションのログアウト/ログインで確実化。
